Storage Layer Implementation Plan

Build a SQLite-backed persistence layer with migrations, CRUD repositories, and full-text search.

Goal: Persist validated URLs, tags, and collections in SQLite. Provide a synchronous, type-safe API that downstream modules (UI, protocol handlers, search) use without knowledge of the underlying database.
Tech Stack: Rust, rusqlite (bundled), sublime_fuzzy
Approach: Domain-split repositories share a single Connection via the Storage struct. Explicit SQL migrations with a Rust runner. Synchronous API throughout.

FTS5 Virtual Table & Triggers

CREATE VIRTUAL TABLE fts_urls USING fts5(
    canonical_url, title,
    content='urls', content_rowid='id'
);

-- Auto-sync triggers keep fts_urls in sync with urls table
CREATE TRIGGER urls_fts_insert AFTER INSERT ON urls ...
CREATE TRIGGER urls_fts_update AFTER UPDATE ON urls ...
CREATE TRIGGER urls_fts_delete AFTER DELETE ON urls ...

Data Flow

Raw URL string │ └─→ ValidatedUrl::parse(input) │ └─→ Storage::urls().insert(&validated, title) │ └─→ SQLite (urls table + fts_urls index) │ └─→ Search or UI query returns UrlRecord

Search Strategy

Tier 1 — FTS5: Broad text filtering. Queries like example match example.com and titles containing "example". Used for the initial filter.
Tier 2 — In-Memory Fuzzy: On startup, load all non-archived URL titles and canonical URLs into memory. Score each candidate with sublime_fuzzy::FuzzySearch. Return top-N results sorted by score. Memory footprint < 1MB for thousands of URLs.

Success Criteria

  • Storage::open creates or migrates the database on first run.
  • UrlRepository::insert accepts a ValidatedUrl and returns an ID.
  • Duplicate canonical URLs are rejected with ConstraintViolation.
  • FTS search returns relevant results for title and URL text.
  • Archive/pin state changes persist and are reflected in list queries.
  • Tags and collections can be created, attached, detached, and listed.
  • All repository operations return typed errors, never panic.
