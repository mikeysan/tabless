Overview & Goal

Build the src/protocol/ subsystem that registers tabless:// as a custom URL scheme on the host OS. When another app invokes tabless://open?url=..., the app starts (or reuses a running instance), extracts the embedded URL, validates it via ValidatedUrl::parse, stores it in SQLite via Storage::urls::insert, and exits silently.

Single-instance behaviour: The first instance binds an IPC socket and blocks accepting URLs. Subsequent instances connect, send the URL, and exit immediately. No duplicate databases, no race conditions.

Architecture

  +-------------------+     +-------------------+     +-------------------+
  |   Another App     |     |   tabless binary  |     |   tabless binary  |
  |  (already running)  |     |  (first instance)  |     | (second instance) |
  +---------+---------+     +---------+---------+     +---------+---------+
            |                           |                           |
            | tabless://open?url=...    |                           |
            +--------------------------->+                           |
            |                           | 1. Parse protocol URL       |
            |                           | 2. Validate URL           |
            |                           | 3. Store in SQLite        |
            |                           | 4. Bind IPC socket        |
            |                           | 5. Block accepting URLs   |
            |                           |                           |
            |                           | <--------+                |
            |                           |   accept_url()            |
            |                           |   handle_url()            |
            |                           |   store in SQLite         |
            |                           +-------->                |
            |                           |                           |
            |                           |       +-------------------+
            |                           |       | 1. Try connect    |
            |                           |       | 2. Send URL       |
            |                           |       | 3. Exit           |
            |                           |       +-------------------+

The protocol handler sits between the OS launcher and the existing Storage / ValidatedUrl subsystems. It does not touch the launcher module.

Key Types

ProtocolConfig

pub struct ProtocolConfig {
    pub scheme: &'static str,      // "tabless"
    pub binary_path: PathBuf,         // path to the tabless executable
    pub data_dir: PathBuf,            // platform-specific data directory
}

impl ProtocolConfig {
    pub fn socket_path(&self) → PathBuf {
        self.data_dir.join("tabless.ipc")
    }
}

IPC Protocol

Simple line-based protocol over interprocess local sockets (Unix on Linux/macOS, named pipes on Windows).

# Client sends one line
URL:https://example.com\n

# Server reads one line, strips prefix, stores URL
# One URL per connection, one line per message
// IpcServer — binds socket, accepts connections
impl IpcServer {
    pub fn bind(socket_path: &Path) → Result;
    pub fn accept_url(&self) → Result;
}

// IpcClient — connects to existing server, sends URL
impl IpcClient {
    pub fn connect(socket_path: &Path) → Result;
    pub fn send_url(&mut self, url: &str) → Result<(), ProtocolError>;
}

Platform Registration

Each platform registers tabless:// differently. The register-protocol subcommand handles all three.

Linux

  • Create ~/.local/share/applications/tabless.desktop
  • Set MimeType=x-scheme-handler/tabless;
  • Run xdg-mime default tabless.desktop x-scheme-handler/tabless

macOS

  • Write ~/Applications/Tabless.app bundle wrapper
  • Add CFBundleURLTypes to Info.plist
  • Wrapper script calls the real binary
  • Register via lsregister -f

Windows

  • Write registry keys under HKEY_CLASSES_ROOT\tabless
  • Set URL Protocol value
  • Set shell\open\command to binary path
  • Use reg add command
# Register the protocol on any platform
$ tabless register-protocol
Protocol registered successfully.

Testing Strategy

Unit Tests (per module)

  • parse.rs: 6 tests — valid, encoded, wrong scheme, wrong path, missing param, empty param
  • ipc.rs: roundtrip test — server thread + client send + assert received URL matches

Integration Test

  • Spawn server thread: bind IPC socket, store first URL via Storage, accept one forwarded URL, store it too
  • Connect client, send second URL, exit
  • Verify both URLs exist in in-memory SQLite database

Note on blocking tests: The integration test uses a bounded server (accepts exactly one forwarded URL then exits). The production ProtocolHandler::run_server loops forever. Do not test the infinite loop directly — test the components that compose it.

Error Types

Every failure path returns a typed ProtocolError. No panics, no unwraps in production code.

InvalidUrl { reason: String } — Malformed protocol URL, wrong scheme/path, missing query param
UrlValidationFailed { source: UrlValidationError } — Embedded URL rejected by ValidatedUrl
StorageFailed { source: StorageError } — SQLite insert failed
IpcBindFailed { reason: String } — Could not bind local socket
IpcConnectFailed { reason: String } — Could not connect to existing instance
RegistrationFailed { platform: String, reason: String } — OS registration command failed
AlreadyRegistered — Protocol already registered on this system
